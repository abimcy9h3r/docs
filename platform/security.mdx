# Security

ForLarge is designed with security at every layer — from payment verification to content delivery.

## Payment Verification

ForLarge **never trusts client-reported payments**. Every transaction is verified server-side:

<Steps>
  <Step title="Buyer Signs Transaction">
    The buyer signs a transaction in their wallet, sending USDC to the creator.
  </Step>
  <Step title="Transaction Broadcast">
    The signed transaction is broadcast to the blockchain network.
  </Step>
  <Step title="Server-Side Verification">
    ForLarge's servers independently query the blockchain to verify:
    - Transaction exists and is confirmed
    - Correct USDC amount was transferred
    - Payment was sent to the correct creator wallet
    - Platform fee was included
  </Step>
  <Step title="Download Issued">
    Only after server-side verification passes does ForLarge issue a download link.
  </Step>
</Steps>

### Solana Verification

ForLarge uses Helius RPC to query Solana transactions. The server parses token transfer instructions and verifies amounts and recipients match the expected values.

### Injective Verification

ForLarge queries the Injective LCD REST API to retrieve transaction details. MsgSend messages are parsed to verify USDC (Peggy denomination) transfers.

## Download Security

Digital products are protected with multiple security layers:

- **Encrypted tokens** — download URLs use cryptographically random tokens
- **Time-limited access** — tokens expire after a configurable period
- **Single-use enforcement** — tokens are invalidated after download
- **Server-side validation** — every download request is checked against purchase records
- **No direct file URLs** — product files are never directly accessible

## Authentication

ForLarge supports two authentication methods:

### Email + Password
- Passwords are hashed with bcrypt (via Supabase Auth)
- Email verification is required

### Google OAuth
- Server-side OAuth flow — authorization happens entirely on ForLarge's domain
- No third-party auth URLs are exposed to users
- CSRF protection via state tokens
- Tokens exchanged server-side, never in the browser

## Access Control

### Invite-Only Registration
During early access, new accounts require an invite code. This controls growth and prevents spam.

### Row-Level Security (RLS)
The database uses Supabase Row-Level Security policies to ensure:
- Users can only modify their own profiles and products
- Payment records are immutable after creation
- Admin endpoints require admin role verification

### Admin Protection
Admin routes (`/admin/*`) and API endpoints (`/api/admin/*`) verify the user's admin status server-side before processing any request.

## Infrastructure Security

- **HTTPS everywhere** — all traffic is encrypted in transit
- **Supabase** — database and auth hosted on SOC 2 Type II compliant infrastructure
- **Environment variables** — all secrets and API keys are stored as environment variables, never in code
- **No client-side secrets** — service role keys and API secrets are only used server-side

## Reporting Vulnerabilities

If you discover a security vulnerability, please report it responsibly by contacting us directly. Do not disclose security issues publicly.
